# Декомпозиция Эпика
## Цель
Реализовать: добавить на сайт интернет-магазина персональную страницу поздравления пользователя с днем рождения.
## Допущения
- Внутреннее устройство проектов пока не подключено.
- Figma пока недоступна, макеты считаются описанными текстом.
- Jira пока недоступна, задачи сформированы в Markdown.
- Уровень разработчиков: senior.
## Вопросы До Стартa
- Какие точные состояния экранов есть в макетах: loading, empty, error, success?
- Нужны ли feature flags и постепенное включение?
- Есть ли обратная совместимость API для старых мобильных версий?
- Какие аналитические события нужно отправлять?
## Порядок Работ
1. Уточнить контракт API и состояния UI.
2. Подготовить backend-контракт или mock contract.
3. Реализовать frontend/mobile независимо от backend через mock/stub, если контракт утвержден.
4. Подключить реальные API.
5. Провести интеграционную проверку и подготовить release notes.
## Backend
### Подготовить API-контракт для birthday-commerce сценария

- Цель: Описать request/response, ошибки, права доступа и миграции данных.
- Зависимости: Бизнес-правила и список состояний из макетов.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.
- Как тестировать: Unit-тесты на валидацию, contract tests, проверка ошибок.
- Слияние: Можно слить без ручного QA, если это только контракт/OpenAPI без runtime-логики.
### Реализовать backend-логику для birthday-commerce сценария

- Цель: Добавить endpoint/service/use-case и обработку бизнес-правил.
- Зависимости: Утвержденный API-контракт.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.
- Как тестировать: Unit + integration tests, проверка прав доступа и негативных сценариев.
- Слияние: Требуется QA или интеграционная проверка, если меняется runtime-поведение.
## Frontend Web
### Собрать UI-состояния для birthday-commerce сценария

- Цель: Реализовать экран/компонент по словесному описанию макетов.
- Зависимости: Список состояний, тексты, роли пользователя, API contract или mock data.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.
- Как тестировать: Storybook/скриншотная проверка или ручная проверка состояний.
- Слияние: Можно слить в ветку эпика без QA, если компонент изолирован и работает на mock data.
### Подключить web UI к данным

- Цель: Интегрировать UI с API, обработать loading/error/empty/success.
- Зависимости: Готовый backend endpoint или stable mock server.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.

## QA И Аналитика
### Подготовить тестовую матрицу

- Цель: Покрыть happy path, ошибки, пустые состояния, права доступа и регрессии.
- Зависимости: Финальный список задач и API contract.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.
- Как тестировать: Сам документ тестируется ревью команды.
- Слияние: Можно слить без QA.
### Описать аналитические события

- Цель: Зафиксировать события, параметры, точки отправки и владельца дашборда.
- Зависимости: Пользовательский сценарий и продуктовые метрики.
- Критерии готовности: реализация соответствует контракту, покрыты основные состояния, нет скрытых блокировок для других ролей.
- Как тестировать: Проверка в debug/log режиме.
- Слияние: Можно слить без QA, если это только спецификация.
